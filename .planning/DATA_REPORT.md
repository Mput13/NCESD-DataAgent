## 1. Короткий вывод

У нас есть два разных класса локальных данных:

1. **Росстат/ЕМИСС (`fedstatru`)** - российская официальная статистика, много ведомственных и региональных показателей, богатые русскоязычные метаданные, но сами таблицы в основном лежат в "широком" формате с колонками `column00`, `column01`, ... и строкой-заголовком внутри данных. Для надежного извлечения нужен отдельный нормализатор.
2. **World Bank (`wb`)** - международная статистика, 29 470 индикаторов и 296 стран/агрегатов, очень удобный единый Parquet-формат: `indicator_id`, `country_id`, `date`, `value`. Это лучший источник для быстрых сравнительных запросов по странам.

Отдельно есть **живой API репозитория НЦСЭД на CKAN**. Он сейчас возвращает 53 799 наборов данных и должен быть не "бонусом", а третьим источником: через него можно искать полный каталог, получать свежие ресурсы, видеть форматы и ссылки на файлы. Локальные дампы полезны для скорости и воспроизводимости; API полезен для полноты и свежести.

Главный архитектурный вывод: ассистенту нужен **каталог метаданных отдельно от числовых таблиц**. LLM ищет и маршрутизирует по каталогу, а числа извлекает детерминированный код из Parquet/JSONL/API-ресурсов.

### 3.5. Как выглядят Parquet-файлы ЕМИСС

Большинство Parquet не являются аккуратным long-format. Это широкие таблицы:

- колонки технические: `column00`, `column01`, ... или `column0`, `column1`, ...;
- первая строка содержит названия измерений и годы;
- следующие строки содержат значения измерений и значения по годам;
- годовые колонки являются отдельными столбцами;
- значения часто надо "melt" в формат `dimensions + year + value`.

Пример `57319.parquet` - ВВП в рыночных ценах по СНС 2008:

```text
rows: 31
columns: column00..column17
row 0:
  column00 = "Вид цены"
  column01 = "Классификатор объектов административно-территориального деления (ОКАТО)"
  column02 = "Единица измерения"
  column03 = "Период"
  column04..column17 = 2011..2024
row 1:
  Вид цены = "8 Постоянные цены 2011 года"
  ОКАТО = "643 Российская Федерация"
  Единица измерения = "385 миллион рублей"
  Период = "54 I квартал"
  2011 = 13386065.5
  2012 = 14114583.9
  ...
```

Пример `40578.parquet` - ВВП России по ППС:

```text
rows: 3
columns: column00..column31
years: 1995..2023
dimensions:
  Классификатор объектов административно-территориального деления (ОКАТО)
  Единица измерения
  Период
```

Пример `61028.parquet` - индексы потребительских цен на отдельные товары и услуги:

```text
rows: 10 592 038
columns: column0..column8
years: 2021..2025
dimensions:
  Виды товаров и услуг
  ОКАТО
  Единица измерения
  Период
```

`clean_jsonl/{code}.jsonl.gz` уже нормализован примерно так:

```json
{
  "Классификатор объектов административно-территориального деления (ОКАТО)": "643 Российская Федерация",
  "Единица измерения": "383 рубль",
  "Период": "115 январь",
  "Социально-демографические группы населения": "9 Все население",
  "year": "1992",
  "value": "635.0"
}
```

Это почти идеальный формат для детерминированного извлечения, но таких файлов только 84. Их можно использовать для MVP и как образец нормализации остальных Parquet.

### 3.7. Как работать с ЕМИСС

Рекомендуемый pipeline:

1. **Не полагаться на `metadata.jsonl`.** Он фактически неполный.
2. **Собрать каталог `fedstat_catalog` из `metdata.csv` + `metadata/{code}.json` + фактического наличия `parquet/{code}.parquet` и `clean_jsonl/{code}.jsonl.gz`.**
3. **Индексировать только метаданные**, а не сами числовые таблицы:
   - `code`
   - `name`
   - `url`
   - единицы измерения
   - временной диапазон
   - признаки/измерения
   - методологические пояснения
   - ведомство
   - подразделение
   - наличие clean JSONL
   - наличие Parquet
   - число строк
4. **Для извлечения данных выбирать стратегию:**
   - если есть `clean_jsonl`, читать его напрямую;
   - если есть только Parquet, нормализовать широкий формат:
     - прочитать первую строку как внутренний заголовок;
     - разделить dimension-колонки и year-колонки;
     - привести в long-format: `code`, `indicator_name`, `dimensions`, `year`, `value`, `unit`, `source_url`;
     - вычистить коды в начале значений, например `643 Российская Федерация`;
     - сохранить результат как derived cache.
5. **Сравнивать итоговую строковую мощность с `rows` из `metdata.csv`.** Это поможет ловить ошибки парсинга.
6. **Для больших файлов не читать все в память.** `61028.parquet` имеет больше 10 млн строк; нужен PyArrow/DuckDB/Polars с фильтрацией.

### 4.4. Данные World Bank в Parquet

Это самый удобный слой в обоих дампах. Каждый файл - один индикатор, например:

```text

```

Схема единая:

| Колонка | Тип | Смысл |
|---|---|---|
| `indicator_id` | string | код индикатора |
| `indicator_name` | string | название |
| `country_id` | string | двухбуквенный/служебный код WB |
| `country_name` | string | страна или агрегат |
| `countryiso3code` | string | ISO-3, например `RUS`, `KAZ` |
| `date` | int64 | год |
| `value` | double | значение, может быть null |
| `unit` | string | почти всегда null |
| `obs_status` | string | статус наблюдения |
| `decimal` | int64 | число знаков округления |

1. **Собрать `worldbank_catalog` из `indicators.json`, `metadata.json`, `metadata.jsonl`, `sources.json`, `countries.json` и списка Parquet-файлов.**
2. **Индексировать `id`, `name`, `sourceNote`, `sourceOrganization`, `topics`, `source`.**
3. **Извлекать данные прямой фильтрацией:**
   - выбрать `indicator_id`;
   - открыть `parquet/{indicator_id}.parquet`;
   - фильтровать `countryiso3code` или `country_id`;
   - фильтровать `date`;
   - убрать `value is null`;
   - вернуть таблицу + metadata + ссылку/описание источника.
4. **Для стран иметь словарь синонимов:**
   - Россия -> `RUS`, `Russian Federation`;
   - Казахстан -> `KAZ`, `Kazakhstan`;
   - США -> `USA`, `United States`;
   - Европа -> набор стран или агрегат, в зависимости от запроса.
5. **Отдельно решать вопрос единиц измерения.** Поле `unit` пустое, значит для вывода надо использовать название индикатора и при необходимости парсить unit из него.

World Bank - лучший кандидат для MVP-запросов вида:

- "Сравни динамику ВВП стран Европы за 2020-2025";
- "Как связаны урбанизация и рождаемость по миру";
- "Инфляция по странам БРИКС";
- "Население, ВВП, безработица по странам".

Он хуже для:

- российских региональных запросов;
- детальных ведомственных показателей РФ;
- двусторонней торговли Россия-Казахстан в товарной детализации.

### 5.2. Что дает API

API НЦСЭД - это не замена локальным дампам, а слой discovery и freshness:

- найти наборы данных по русскому запросу;
- получить название, описание, организацию, теги, лицензию;
- получить список ресурсов и прямые ссылки на файлы;
- проверить, есть ли более свежая версия, чем в локальном дампе;
- добрать источники, которых нет локально;
- показать пользователю "след": какие источники нашлись и почему выбран конкретный.

По данным веб-страницы репозитория, наборы данных доступны в форматах вроде `csv.gz`, `parquet`, `xls.zip`, `HTML`; для некоторых внешних источников встречаются `XLSX`, `JSON`, `PARQUET`, `XML`, `OTHER`.

### 5.3. Как с ним работать

Минимальный набор методов для adapter-а:

```text
search_packages(query, rows=10, start=0, fq=None)
  -> package_search

get_package(id_or_name)
  -> package_show

search_resources(query)
  -> resource_search

download_resource(resource_url)
  -> requests/urllib + cache
```

Рекомендуемая логика:

1. Сначала искать локально в `fedstat_catalog` и `worldbank_catalog`.
2. Если локально нет уверенного результата или нужно проверить свежесть - идти в CKAN `package_search`.
3. Для найденного пакета получать resources.
4. Если ресурс уже есть локально, использовать локальный файл.
5. Если локально нет, скачать ресурс в cache и обрабатывать тем же deterministic extractor-ом.
6. Всегда сохранять provenance:
   - `package_id/name`
   - `resource_id/name`
   - `resource_url`
   - `organization`
   - `license`
   - `metadata_modified`
   - дата обращения к API.

### 5.4. Риски API

- API живой: результаты, количество пакетов и metadata_modified могут меняться.
- CKAN search возвращает много нерелевантного при широких запросах; нужен ранжировщик поверх `package_search`.
- `package_search` не гарантирует, что нужный год/страна реально есть внутри файла; после discovery нужна проверка фактической таблицы.
- Некоторые пакеты могут быть `isopen=false` или иметь отсутствующую/неясную лицензию; это надо показывать в следе выбора источника.
- Для массовой загрузки нужно кэширование и ограничение запросов.

## 6. Что это значит для ассистента

### 6.1. Роли источников

| Источник | Роль | Когда использовать |
|---|---|---|
| Локальный ЕМИСС | российские официальные показатели, регионы, ведомственные данные | ВВП РФ по Росстату, ИПЦ, доходы, зарплаты, рождаемость, региональные показатели |
| Локальный World Bank | международные сравнения и макроиндикаторы | страны, группы стран, BRICS, Европа, динамика 1960-2024/2025 |
| НЦСЭД CKAN API | широкий реестр и свежие ресурсы | когда локального источника мало, нужен полный каталог или прямая ссылка на ресурс |
| Live World Bank API | fallback/обновление WB | если локальный Parquet отсутствует или хочется проверить актуальность |
| Другие открытые API | расширение | ВТО/UN Comtrade/UNESCO/UNIDO для задач, которых нет в текущих двух дампах |

### 6.2. Что индексировать

Индексировать надо **карточки источников**, а не числовые таблицы. Карточка должна содержать:

```text
api_resource_urls

```

`availability_flags`:

```text

```

`quality_flags`:

```text

```

### 6.3. Инструменты агента

Для MCP/function calling нужны не "общие" инструменты, а компактные data-tools:

```text
find_data(query, geography=None, period=None, source=None, limit=10)
  Ищет в каталоге и возвращает кандидаты с объяснением.

inspect_dataset(dataset_id, source)
  Возвращает паспорт: schema, coverage, units, dimensions, examples, source links.

get_table(dataset_id, source, filters)
  Детерминированно извлекает таблицу из локального файла/API resource.

build_dataset(spec)
  Собирает итоговый CSV/Excel/Parquet из одного или нескольких get_table.

explain_no_data(query, attempted_sources)
  Формирует честное объяснение, где искали и чего не хватило.
```

LLM должна:

- понять запрос;
- выбрать источник/инструмент;
- сформулировать уточняющий вопрос;
- объяснить пользователю результат.

Код должен:

- читать файлы;
- фильтровать строки;
- считать метрики;
- проверять наличие данных;
- формировать CSV/Excel.

### 6.4. Поиск: не начинать только с embeddings

На этих данных нужен гибрид:

1. Нормализация запроса:
   - русский/английский;
   - синонимы: ВВП/GDP, инфляция/CPI, рождаемость/fertility;
   - страны: Россия/Russian Federation/RUS;
   - периоды: 2015-2024, 2020-2025.
2. Лексический поиск по `title/name/sourceNote/methodology/tags`.
3. Фильтры по source, географии, времени, единицам, наличию данных.
4. Rerank кандидатов с учетом:
   - exact match в названии;
   - официальный источник;
   - наличие локального Parquet;
   - покрытие периода;
   - соответствие географии;
   - единицы измерения.
5. Embeddings/File Search можно добавить после baseline, но не как единственный механизм.

Практический пример: поиск по слову `инфляция` в ЕМИСС дает мало прямых попаданий, а `потребительских цен` находит нужный класс показателей. Значит, без словаря синонимов и экономических соответствий поиск будет шумным.

## 8. Рекомендуемый MVP-скоуп

### Минимально полезный MVP

Сделать end-to-end на двух источниках:

1. **World Bank local** как простой, надежный источник.
2. **Fedstat local** на ограниченном наборе показателей:
   - ВВП `57319`;
   - ВВП ППС `40578`;
   - ИПЦ/инфляция `33568` или `61028`;
   - рождаемость `30973`;
   - доходы/зарплаты из нескольких выбранных кодов.
3. **НЦСЭД CKAN API** как live discovery и источник ссылок.

### Почему так

- World Bank быстро дает красивый международный demo без тяжелой нормализации.
- ЕМИСС демонстрирует ценность для российских данных и соответствие кейсу НЦСЭД.
- API показывает, что система не замкнута на скачанном архиве и умеет ориентироваться в полном репозитории.

### Что не делать в первую очередь

- Не нормализовать все 7 328 Parquet ЕМИСС сразу. Это может съесть время.
- Не строить embeddings по числовым таблицам.
- Не заставлять LLM читать CSV/Parquet напрямую.
- Не обещать bilateral trade Россия-Казахстан без подключения дополнительного источника.

## 11. Риски и открытые вопросы

| Риск | Почему важен | Что делать |
|---|---|---|
| Неполный `metadata.jsonl` ЕМИСС | README обещает агрегат, но фактически там 4 строки | Строить каталог из `metdata.csv` и `metadata/*.json` |
| Широкий Parquet ЕМИСС | Нельзя просто фильтровать `year/value` | Написать нормализатор wide -> long |
| Только 84 `clean_jsonl` | Удобный формат покрывает малую часть | Использовать как sample/MVP, затем расширять нормализатор |
| Расхождения indicator list vs Parquet | 1 480+ кодов без данных/метаданных | В каталоге хранить availability flags |
| World Bank `unit` пустой | Пользователь должен видеть единицы | Извлекать unit из названия/sourceNote |
| 2025 в WB часто null | Запросы 2025 могут не закрываться | Проверять non-null values и объяснять |
| CKAN search шумный | `ВВП` вернул 3 310 результатов | Нужен rerank и фильтры |
| Bilateral trade не покрыт | Один из примеров ТЗ про Россия-Казахстан | Планировать отдельный источник: CKAN, ВТО, UN Comtrade |
| Лицензии/открытость | Не все пакеты могут быть `isopen=true` | Проверять `license`, `isopen`, resource URL |
| Свежесть данных | локальный дамп может устаревать | API metadata_modified + cache refresh |

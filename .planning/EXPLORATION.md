## Сквозной вопрос 1: Нужен ли agentic workflow?

### Вариант A: Линейный пайплайн (без агента)

```
Запрос → classify() → search() → extract() → format() → ответ
```

- **+** Простой, предсказуемый, легко дебажить
- **−** Не умеет уточнять, не адаптируется к ошибкам, жёсткая последовательность

### Вариант B: Агент с инструментами (ReAct / tool-use loop)

```
Запрос → LLM решает что делать → вызывает tool → смотрит результат → решает снова → ...
```

- **+** Адаптивный: сам решает когда уточнить, когда искать ещё
- **+** Естественно поддерживает уточняющий диалог
- **+** Соответствует ТЗ («агент, а не вопрос-ответная LLM»)
- **−** Менее предсказуемый, дороже по токенам

### Вариант C: Гибрид — детерминированный скелет + LLM в узлах

```
classify(LLM) → search(deterministic) → extract(deterministic) → synthesize(LLM)
```

- **+** Контроль потока, но LLM для «умных» шагов
- **+** Детерминированные шаги не тратят токены и не галлюцинируют
- **−** Теряем адаптивность (агент не может «передумать»)

> **Вердикт:** Вариант B (агент) — скорее всего MUST, потому что ТЗ явно говорит про агента. Но внутри агента tools должны быть детерминированными (вариант C внутри B).

## Сквозной вопрос 2: Оркестрация агента

| Вариант | Плюсы | Минусы | Для нас |
|---------|-------|--------|---------|
| **Custom loop** | Полный контроль, минимум зависимостей | Нужно писать парсинг, retry, трассировку | ✅ LIKELY |
| **Yandex AI Studio Responses API** | Нативная интеграция, function calling | Привязка к экосистеме, доки ограничены | ✅ LIKELY |

| **LangChain / LangGraph** | Экосистема, retrievers | Тяжёлый, магия, overengineered | ❌ Скорее нет |

> **Ключевой вопрос:** Совместим ли Yandex AI Studio с OpenAI SDK? → Мастер-класс 9 мая.

## Этап 0: Предобработка и индексация данных (offline)

> Подготовить данные ОДИН РАЗ, чтобы агент работал с обработанным вариантом.

### Что индексируем

| Источник | Объём | Формат |
|----------|-------|--------|
| Росстат (ЕМИСС) | ? записей | metadata.jsonl + parquet/ |
| World Bank | 29K+ индикаторов, 296 стран | indicators.json + parquet/ |
| НЦСЭД CKAN | ~54K наборов | JSON API (live) |

### Вариант 0A: Каталог метаданных без эмбеддингов

Структурированный JSON: название, описание, поля, период, страны.

- **+** Нулевой setup, LLM может читать если помещается
- **−** 29K индикаторов не влезут в контекст LLM
- **Когда работает:** После предварительной фильтрации до ~50-100 записей

### Вариант 0B: Full-Text Search (BM25 / DuckDB FTS) ✅ MUST

Keyword-поиск по названиям и описаниям индикаторов.

```python
# DuckDB FTS
duckdb.sql("""
    PRAGMA create_fts_index('indicators', 'id', 'name', 'description');
    SELECT * FROM indicators 
    WHERE fts_main_indicators.match_bm25(id, 'ВВП рост') IS NOT NULL;
""")
```

- **+** Быстрый, точный для конкретных терминов, zero-dependency
- **+** Отлично с кодами, аббревиатурами, точными названиями
- **−** Не понимает синонимы («GDP» ≠ «ВВП»)
- **Оценка:** ✅ **MUST как baseline**

### Вариант 0C: Векторный индекс (embeddings + FAISS/ChromaDB) ⚠️ LIKELY

```python
# Offline: один раз
embeddings = model.encode([ind["description"] for ind in indicators])
faiss_index.add(embeddings)
# Runtime:
results = faiss_index.search(model.encode("рост экономики"), k=20)
```

- **+** Понимает семантику, синонимы, перефразирования
- **−** Lossy: теряет точные значения, коды
- **−** Нужна multilingual модель эмбеддингов (какая?)
- **Оценка:** ⚠️ **LIKELY** — но НЕ как единственный метод

### Вариант 0D: Гибридный поиск (BM25 + Vector + RRF) ✅ ИДЕАЛ

Оба метода параллельно → Reciprocal Rank Fusion → опционально reranker.

- **+** Лучшее из двух миров
- **−** Сложнее в реализации
- **Оценка:** ✅ **Идеальный вариант если есть время**

### Вариант 0E: Yandex AI Studio Vector Stores API ⚠️ OPTIONAL

Загрузить файлы в Yandex Cloud → их managed поиск.

- **+** Не нужно настраивать инфраструктуру
- **−** Чёрный ящик, зависимость от облака

### Матрица решений: Индексация

| Критерий | 0B: FTS | 0C: Vector | 0D: Hybrid | 0E: Yandex |
|----------|---------|------------|------------|------------|
| Сложность | Низкая | Средняя | Высокая | Низкая |
| Точные термины | ⭐⭐⭐ | ⭐ | ⭐⭐⭐ | ⭐⭐ |
| Семантика | ⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| Hackathon-ready | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ |

> **Рекомендация:** Начать с **0B (FTS)** → если недостаточно, добавить **0C** → получить **0D**.

## Этап 1: Понимание запроса (NLU)

### 1A: LLM-классификация через промпт ✅ MUST

```
System: Классифицируй тип запроса: simple/comparative/research/derived/ambiguous/no_data.
Извлеки: страны, период, индикатор, ракурс.
```

### 1B: Structured output (JSON / function calling) ✅ MUST

LLM возвращает JSON `{type, countries, period, indicator}` — парсится программно.

### 1C: Уточняющий диалог ✅ MUST

Тип=ambiguous → агент задаёт вопросы → пользователь отвечает → re-classify.
Обязательно по ТЗ (тип «неоднозначный»).

### 1D: Few-shot примеры в промпте ✅ MUST

5-10 примеров «запрос → разбор» в system prompt. Дёшево и сильно повышает качество.

## Этап 2: Поиск релевантных источников

### Стратегия по типу источника

| Источник | Стратегия | Когда |
|----------|-----------|-------|
| Росстат (локальный) | FTS/Vector по metadata.jsonl | Всегда первый |
| World Bank (локальный) | FTS/Vector по indicators.json | Всегда первый |
| НЦСЭД CKAN (API) | HTTP package_search | Если локально не нашли |
| World Bank API (live) | HTTP indicators API | Fallback |

### 2A: Единый объединённый индекс ✅ LIKELY

Нормализовать metadata обоих источников → один индекс → один запрос.

### 2B: Раздельные поиски + merge ✅ LIKELY

Параллельный поиск по каждому → объединение. Более гибкий, легче добавлять источники.

### 2C: LLM-маршрутизация (Router) ⚠️ OPTIONAL

LLM решает КУДА искать: «ВВП России» → Росстат, «GDP BRICS» → World Bank.

### 2D: Reranking результатов ⚠️ OPTIONAL

Cross-encoder или LLM-reranker для top-N. Полировка.

## Этап 3: Извлечение данных

> **ЖЁСТКОЕ ОГРАНИЧЕНИЕ:** Числа извлекает КОД, не LLM.

### 3A: DuckDB SQL ✅ MUST (основной инструмент)

```python
result = duckdb.sql("""
    SELECT year, value FROM 'worldbank/NY.GDP.MKTP.CD.parquet'
    WHERE country_code = 'RUS' AND year BETWEEN 2015 AND 2024
""").df()
```

- Читает Parquet нативно, мощные агрегации, LLM хорошо генерирует SQL

### 3B: Pandas ⚠️ OPTIONAL

Для пост-обработки, не для основного извлечения. LLM генерирует менее надёжный pandas-код.

### 3C: LLM генерирует код (Text-to-SQL) ✅ MUST

Агент получает схему → генерирует SQL → выполняет → возвращает DataFrame. Ядро системы.

### 3D: Self-healing execution ✅ LIKELY

SQL упал → ошибка обратно LLM → пере-генерирует. Стандартная практика.

### 3E: Предварительный просмотр схемы ✅ MUST

`DESCRIBE TABLE` или head(5) перед генерацией — LLM видит реальную структуру.

### 3F: Sandbox для кода ⚠️ OPTIONAL

Для хакатона достаточно READ-ONLY доступа.

## Этап 5: Формирование ответа

### 5A: LLM-синтез текста ✅ MUST

LLM получает данные + контекст → текстовый ответ. НЕ придумывает числа.

### 5B: Структурированные артефакты ✅ MUST (по ТЗ)

CSV/Excel + Python-скрипт сборки + JSON метаданные + ссылки на источники.

### 5C: «След» работы (trace) ✅ MUST (по ТЗ)

Какие шаги, какие источники нашёл/отверг, какой код выполнил.

### 5D: Визуализация (графики) ⚠️ OPTIONAL
